1. 总览：先建立正确心智模型

一句话理解

这个仓库实现的是一个 事件驱动的 Agent Runtime：Runner 管运行时，Agent 产事件，Session 存历史和状态，LLM Flow 调模型和工具，Plugin 做横切扩展，REST/SSE 把事件流给前端。

最重要的抽象

Agent.Run(ctx) 返回的是 iter.Seq2[*session.Event, error]，不是字符串。也就是说，整个系统是围绕事件流设计的：用户输入、模型输出、工具调用、工具结果、状态更新、转交都被表示成事件。

为什么这样设计

因为真实 Agent 不只是“问模型拿答案”。它需要多轮工具调用、人工确认、跨 Agent 转交、流式输出、状态更新、审计追踪、文件制品和长期记忆。这些都必须有统一的数据协议承载。

读代码主线

推荐从 runner/runner.go 开始，看一次请求如何进入系统；再看 internal/llminternal/base_flow.go 理解模型-工具循环；最后看 session/inmemory.go 理解事件如何保存。

本次修改：已经在关键源码中新增中文业务注释，并新增多个 zz_cn_walkthrough.go 包级走读文件，帮助你从源码内部理解模块职责。

源码走读

  1. Runner.Run 是入口。它不会直接返回最终文本，而是返回一个事件迭代器。
  2. 先通过 session.Service 读取会话；如果开启 AutoCreateSession，会话不存在时自动创建。
  3. findAgentToRun 根据历史事件判断本轮由谁接手。这样工具确认、函数响应、子 Agent 连续对话不会丢上下文。
  4. Runner 创建 InvocationContext，把 Session、Artifact、Memory、Agent、用户输入、RunConfig 装进去。
  5. appendMessageToSession 把用户消息先写入 Session，使后续 prompt 构造时能看到本轮输入。
  6. Runner 消费 agentToRun.Run(ctx) 产生的事件；非 partial 事件会调用 SessionService.AppendEvent 持久化。
设计重点：Runner 自己不做模型调用，也不做业务工具逻辑。它只负责把运行时上下文装配好，并保证事件被正确分发、保存、返回。

4. Session 管理：如何实现会话、历史、状态

Session 是 Agent 系统里的“事实账本”。如果没有 Session，Agent 就只能做单轮问答；有了 Session，系统才能支持多轮上下文、工具结果回灌、状态变量、跨 Agent 协作和恢复执行。

核心对象

对象作用源码
Session一条会话线程，包含 ID、AppName、UserID、State、Events、更新时间。session/session.go
Event一次原子运行事实：用户消息、模型回复、工具调用、工具结果、状态变更、转交等。session/session.go
StateKV 状态，可被指令模板、工具、回调读取。session/session.go
Service会话存储接口，支持 Create/Get/List/Delete/AppendEvent。session/service.go
inMemoryService内存实现，最适合学习状态合并和事件追加逻辑。session/inmemory.go

状态作用域

Key 前缀作用域业务含义是否长期保存
app:应用级同一 app 下所有用户/会话共享，例如全局配置。是
user:用户级同一 app + user 的多条会话共享，例如用户偏好。是
无前缀Session 级当前会话私有，例如本轮任务进度。是
temp:Invocation 临时级只在当前用户请求链路内使用，例如临时中间变量。否，追加事件前会裁剪

AppendEvent 的业务含义

AppendEvent 不是简单 append 数组，它同时做四件事：

  1. 拒绝 nil session / nil event，避免破坏会话账本。
  2. 忽略 Partial 流式片段，只保存最终事件，防止历史被大量半截文本污染。
  3. 调用 sess.appendEvent(event) 更新当前返回给 Runner 的 session 副本。
  4. 把事件复制进存储层，并根据 event.Actions.StateDelta 更新 app/user/session 状态。
注意：内存实现用 sync.RWMutex 保证并发安全；生产环境如果换成数据库实现，需要保持同样的语义：事件追加和状态更新最好是同一事务。

5. LLM Agent 推理循环：模型如何和工具反复协作

LLM Agent 的真正核心在 internal/llminternal/base_flow.go。它实现的不是“一次模型调用”，而是一个循环：模型可能先回复工具调用，框架执行工具，把工具结果作为 FunctionResponse 加入上下文，再次调用模型，直到最终回复。

RequestProcessors 是什么

DefaultRequestProcessors 是一条 Prompt 构造流水线。它按顺序把基础配置、工具声明、认证/确认请求、Instruction、身份信息、Session 历史、规划内容、代码执行、输出 Schema、Agent 转交工具等塞进同一个 LLMRequest。

重点文件：

  • internal/llminternal/base_flow.go：总流水线。
  • internal/llminternal/contents_processor.go：如何读取 Session.Events 并组装历史上下文。
  • internal/llminternal/instruction_processor.go：如何处理 Instruction / GlobalInstruction / 模板变量。
  • internal/llminternal/outputschema_processor.go：结构化输出如何进入请求。
  • internal/llminternal/agent_transfer.go：Agent 转交工具如何被注入。

什么时候停止循环

Event.IsFinalResponse 决定一个事件是否是最终响应。包含 function_call、function_response、partial、尾部代码执行结果等情况，通常不是最终响应，需要继续循环。

8. 工作流 Agent：顺序、并行、循环

类型行为适合场景源码
SequentialAgent按顺序执行所有子 Agent。固定流水线：需求分析 -> 生成 -> 审查 -> 修复。agent/workflowagents/sequentialagent/agent.go
ParallelAgent并发执行子 Agent，并用 Branch 隔离上下文。多专家并行评审、多方案生成。agent/workflowagents/parallelagent/agent.go
LoopAgent重复执行子 Agent，直到次数结束或 Escalate。迭代改进：写代码 -> 测试 -> 修复。agent/workflowagents/loopagent/agent.go

ParallelAgent 的关键点

并行 Agent 会给每个子 Agent 创建新的 InvocationContext，并设置不同的 Branch。Branch 的意义是隔离同一轮并行任务中的中间历史，避免 A 专家看到 B 专家的中间输出。

源码里还有一个重要细节：子 Agent 把事件发给 Runner 后会等待 ack，确保上游已处理事件再继续，降低并发乱序风险。

11. 你可以迁移到自己 Agent 项目的设计模式

事件驱动，而不是字符串驱动

把模型输出、工具调用、工具结果、状态更新都抽象成 Event。这样系统天然支持回放、审计、恢复、多 Agent 协作。

Runner 做编排，Agent 做执行

Runner 不写业务，Agent 不管 HTTP。职责分离后，你可以替换前端、替换存储、替换模型，而不改 Agent 核心。

上下文对象集中传能力

InvocationContext/ToolContext/CallbackContext 让各层拿到该拿的能力，同时避免函数参数爆炸。

Prompt 构造做成 Pipeline

Instruction、History、Tools、Schema、Memory 都是 RequestProcessor。以后加新能力不是改一坨 prompt 拼接代码，而是加 processor。

工具是结构化协议

工具参数和返回值要有 Schema，工具结果以 FunctionResponse 形式回灌模型，避免自由文本不可控。

插件处理横切问题

日志、审计、缓存、限流、安全检查不要塞进 Agent 业务逻辑，而是做成 Plugin。

最小自研 Go Agent 框架可以按这个骨架做

type Event struct { /* 用户消息、模型消息、工具调用、工具结果、状态变化 */ }
type SessionService interface { Get(...); Create(...); AppendEvent(...) }
type Agent interface { Run(ctx InvocationContext) iter.Seq2[*Event, error] }
type Runner struct { session SessionService; root Agent; plugins []Plugin }
type Tool interface { Name() string; Declaration() Schema; Run(ctx ToolContext, args map[string]any) (map[string]any, error) }
